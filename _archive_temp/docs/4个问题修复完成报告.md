## ✅ 问题3：数据表时间显示不对

### 问题描述
前端显示的时间都是"2026-03-08 08:57:22"，明显错误

### 原因分析
1. SQLite的`CURRENT_TIMESTAMP`返回的是UTC时间
2. API直接返回数据库中的时间，没有转换为北京时间
3. 前端显示的是UTC时间，与北京时间相差8小时

### 解决方案
**修改位置：** `backend/main.py` 第1223行

**旧代码：**
```python
return {"total": total, "tasks": [dict(r) for r in rows]}
```

**新代码：**
```python
# 转换时间为北京时间
from backend.utils.time_utils import timestamp_to_beijing
tasks = []
for r in rows:
    task = dict(r)
    # 转换discovered_at和processed_at为北京时间
    if task.get("discovered_at"):
        task["discovered_at"] = timestamp_to_beijing(task["discovered_at"])
    if task.get("processed_at"):
        task["processed_at"] = timestamp_to_beijing(task["processed_at"])
    tasks.append(task)

return {"total": total, "tasks": tasks}
```

### 效果
- ✅ 自动将UTC时间转换为北京时间（+8小时）
- ✅ 前端显示正确的时间
- ✅ 使用项目统一的时间工具函数

### 验证方法
重启后端后，前端应显示正确的北京时间，例如：
```
2025-03-08 16:57:22  （而不是 2026-03-08 08:57:22）
```

## ✅ 问题4：Miner Agent超时时间显示

### 问题描述
用户认为Miner Agent的超时时间没有从env中获取

### 原因分析
实际上代码是正确的，`settings.llm_timeout`会根据`LLM_MODE`自动选择：
- `LLM_MODE=local` → 使用 `LLM_LOCAL_TIMEOUT`
- `LLM_MODE=remote` → 使用 `LLM_REMOTE_TIMEOUT`

### 验证
查看`backend/services/crawler/question_extractor.py` 第53行：
```python
logger.info(f"   - Timeout: {settings.llm_timeout}s")
```

这里的`settings.llm_timeout`会自动根据模式选择正确的超时配置。

### 当前配置
```bash
LLM_MODE=local
LLM_LOCAL_TIMEOUT=300  # 已修改为300秒
LLM_REMOTE_TIMEOUT=300
```

### 效果
- ✅ 代码逻辑正确，会自动读取对应模式的超时配置
- ✅ 修改问题2后，超时时间已增加到300秒
