## 📋 新架构总结

### 数据流

```
1. Hunter（爬虫）
   输入：URL
   输出：原文
   LLM：❌
   ↓
2. Miner Agent（挖掘）
   输入：原文
   输出：{meta, questions}
   LLM：✅ 使用LLM挖掘
   ↓
3. Knowledge Manager（管理）
   输入：{meta, questions}
   输出：图谱 + 数据库
   LLM：❌ 纯数据操作
   查重：只避免完全重复
   ↓
4. Interviewer Agent（对话）
   输入：用户问题
   输出：面试回答
   LLM：✅ 对话和推理
   检索：RAG自动找相似题目
```

### 组件对比

| 组件 | 性质 | 有LLM | 职责 | 是否查重 |
|------|------|-------|------|---------|
| **Hunter** | 爬虫服务 | ❌ | 抓取原文 | ❌ |
| **Miner Agent** | Agent | ✅ | 挖掘信息 | ❌ |
| **Knowledge Manager** | 管理服务 | ❌ | 数据管理 | ✅ 数据库去重 |
| **Interviewer Agent** | Agent | ✅ | 面试对话 | ❌ (RAG检索) |

## 📊 重构收益

### 1. 架构更清晰

**旧架构：**
- Extractor + Architect混合职责
- 语义查重丢失信息

**新架构：**
- Miner专注提取
- Knowledge Manager专注管理
- 职责单一，易维护

### 2. 提取更智能

**旧方式：**
- 可能使用正则匹配
- 标签不够精确

**新方式：**
- 明确使用LLM语义理解
- 200+个精确技术标签
- 多层级标签（技术+特性）

### 3. 数据更完整

**旧方式：**
- 语义查重合并相似题目
- 丢失不同公司的相似题

**新方式：**
- 只避免完全重复
- 保留所有相似题目
- RAG检索时自然找到

### 4. 配置更统一

**旧配置：**
- Extractor只有temperature
- 缺少max_tokens

**新配置：**
- 所有Agent都有完整配置
- LOCAL/REMOTE统一
- max_tokens避免截断
